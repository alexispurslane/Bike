<h1 id=“label-The+Bike+programming+language”>The Bike programming language<span><a href=“#label-The+Bike+programming+language”>&para;</a> <a href=“#documentation”>&uarr;</a></span></h1>

<p><a href=“christopherdumas.github.io/Bike”>Official Website</a></p>

<p><a href=“Contact”>christopherdumas@me.com“>Contact author</a> or @christopherdumas</p>

<p>The Bike programming language is a programming language that combines Haskell&#39;s currying, function composition, anonymous functions, and array comprehension, with Scala&#39;s packages, classes, and traits, Python&#39;s is and isnt style operators, Scheme&#39;s everything is an expression idea, Ruby&#39;s everything-is-an-object philosophy, and Go&#39;s no paren control structures, with Coffeescript&#39;s infix if and unless. It also has optional no-paren function calls, and immutable arrays are also present. This is mixed in with mixins and Java&#39;s ananymous Classes, as well as JavaScript&#39;s Objects (called hashes) and ES6&#39;s destructuring assignment. It is immutable in everything by default, but in a way that makes it fairly painless to use.</p>

<p>Bike&#39;s motto is:</p>

<blockquote> <p>Not a lot of syntax, but a lot more sugar.</p> </blockquote>

<p>Some code samples are in order:</p>

<h2 id=“label-Roadmap”>Roadmap<span><a href=“#label-Roadmap”>&para;</a> <a href=“#documentation”>&uarr;</a></span></h2>

<p>In the <a href=“TODO_md.html”>TODO.md</a> file is our roadmap to 1.0. It will be updated every major version release. Our versioning system is like this: The whole number (the 1 in 1.0) is incremented each major, or breaking version. The decimal point is incremented by five, and normally for big, but not breaking changes, like adding new operators or shorthand syntax. The Beta version (indicated by +b&lt;number&gt;) is incremented every time a bug is fixed, or a change is being contemplated but I have not fully committed to it.</p>

<h2 id=“label-Contribution”>Contribution<span><a href=“#label-Contribution”>&para;</a> <a href=“#documentation”>&uarr;</a></span></h2>

<h3 id=“label-Issues”>Issues<span><a href=“#label-Issues”>&para;</a> <a href=“#documentation”>&uarr;</a></span></h3>

<p>Please keep issues short and to the point. Please no flaming, or arguments over preferences over syntax. If you make any changes to the syntax, check them with the author first, over email. Also, avoid adding syntax that uses obscure symbols. Also, please no OOP vs. Functional debates. Bike is supposed to make OOP and FP live together. Any debates about them should be about how to make them work together.</p>

<h3 id=“label-Emails+to+author”>Emails to author<span><a href=“#label-Emails+to+author”>&para;</a> <a href=“#documentation”>&uarr;</a></span></h3>

<p>Please keep emails to a minimum, only when concerning major changes/bugs.</p>

<h3 id=“label-Code+Style”>Code Style<span><a href=“#label-Code+Style”>&para;</a> <a href=“#documentation”>&uarr;</a></span></h3>

<p>Please use CamelCase for classes and grammar rules, snake_case for variables, and CAPS_SNAKE_CASE for symbols related to keywords or literals. Please unit test any changes made to the <a href=“Interpreter.html”>Interpreter</a> class or <a href=“Lexer.html”>Lexer</a> class. Also, only comment where things are non-obvous</p>

<h3 id=“label-Commits”>Commits<span><a href=“#label-Commits”>&para;</a> <a href=“#documentation”>&uarr;</a></span></h3>

<p>Please use descriptive commits, but under 80 characters. Always use emoji.</p>
